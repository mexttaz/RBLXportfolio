System 01

Server-authoritative combat framework

Combat is built around a clear trust boundary: the client can request actions, but the server owns outcomes.

The combat stack includes input buffering, recovery windows, hitboxes, damage, stun, block/parry logic, grab checks, knockback, lobby restrictions, and server-created debug indicators. The goal is to make combat feel responsive without letting exploiters control hit results.

BRivalsEngineCore CombatService BlockService GrabService

Responsive Input Without Client Authority

M1 inputs can buffer during recovery for a short window, preserving combo feel while keeping cooldown and recovery validation on the server.

Server-Created Debug Hitboxes

Hitbox indicators are generated by the server, so debugging reflects the same spatial checks used by real gameplay.

Centralized Combat Restrictions

Lobby and tutorial restrictions are handled through dedicated services instead of fragile, repeated checks inside every ability.

Extensible Gameplay Hooks

Combat can connect to progression systems such as pets, perks, rewards, and status effects without turning each ability into a one-off script.

Code Patterns

Portfolio-safe examples of how the systems are structured.

These examples are shortened to show engineering patterns without dumping entire production scripts: input buffering behind server validation, server-created hitbox debug visuals, and queue storage that can run live with MemoryStore while still being testable inside Studio.

BRivalsEngineCore Input Buffering 
-- M1 inputs can queue during recovery
local INPUT_BUFFER_WINDOW = 0.30
local inputBuffer = {}

if CombatService.IsInRecovery(player)
and inputType == "M1" then
  inputBuffer[player.UserId] = {
    inputType = inputType,
    payload = payload,
    standId = standId,
    time = tick(),
  }

  task.spawn(function()
    while inputBuffer[userId]
    and tick() - inputBuffer[userId].time
    < INPUT_BUFFER_WINDOW do
      if not CombatService.IsInRecovery(player) then
        local buffered = inputBuffer[userId]
        inputBuffer[userId] = nil

        BRivalsEngineCore.ProcessAbilityRequest(
          player,
          buffered.inputType,
          buffered.payload
        )

        return
      end

      task.wait(0.016)
    end

    inputBuffer[userId] = nil
  end)
end
CombatService Debug Hitboxes 
-- Debug visuals are created by the server
local HITBOX_INDICATOR_LIFETIME = 0.085

local function makeHitboxIndicatorBase(name)
  local part = Instance.new("Part")
  part.Name = name
  part.Anchored = true
  part.CanCollide = false
  part.CanTouch = false
  part.CanQuery = false
  part.CastShadow = false
  part.Material = Enum.Material.ForceField
  part.Transparency = 0.88
  part.Parent = workspace

  Debris:AddItem(
    part,
    HITBOX_INDICATOR_LIFETIME + 0.05
  )

  return part
end

function CombatService.ShowHitboxRadius(position, radius)
  if not hitboxIndicatorsEnabled then return end

  local part = makeHitboxIndicatorBase(
    "HitboxIndicator_Radius"
  )

  local diameter = math.max(radius * 2, 0.1)
  part.Shape = Enum.PartType.Ball
  part.Size = Vector3.new(diameter, diameter, diameter)
  part.CFrame = CFrame.new(position)
end
QueueStoreAdapter Studio-Safe Queueing 
-- MemoryStore in live servers, fallback in Studio
local fallbackQueues = {}
local memoryStoreAvailable = true

function QueueStoreAdapter.UpsertTicket(ticket)
  local modeId = ticket.QueueMode
  local key = tostring(ticket.UserId)
  local ttl = CompetitiveConstants.Queue.TicketTtlSeconds

  if memoryStoreAvailable then
    local ok, err = pcall(function()
      local map = MemoryStoreService:GetSortedMap(
        getStoreKey(modeId)
      )

      map:SetAsync(
        key,
        ticket,
        ttl,
        ticket.Timestamp or os.time()
      )
    end)

    if ok then
      return true
    end

    memoryStoreAvailable = false
    warn("MemoryStore unavailable; using local fallback", err)
  end

  local queue = getFallbackQueue(modeId)
  queue[key] = {
    Value = ticket,
    ExpiresAt = os.clock() + ttl,
  }

  return true
end

Engineering Decisions

The details that make the work usable on a team.

A hiring page should not just say “I made systems.” It should show how I think: clear authority boundaries, reusable services, predictable module structure, testable infrastructure, and client code that supports real Roblox players on real devices.

Client responsiveness without client-owned damage

Inputs can feel immediate while damage, stun, block, parry, hitbox, and recovery outcomes remain server-owned.

Character kits are content, not architecture rewrites

New stands follow a controller/state/ability structure so the roster can grow without duplicating core combat logic.

Lobby and match servers have different responsibilities

The main place prepares players through queueing, progression, and onboarding while Competitive Alpha handles match runtime.

Match flow accounts for failure cases

Reconnect, AFK, forfeit, spectator, return-to-lobby, cleanup, and reward services exist because real multiplayer sessions break.

Networking is grouped by gameplay domain

Combat, progression, inventory, matchmaking, party flow, competitive state, score, round, spectator, and reward events are separated by purpose.

Studio-safe fallbacks keep development moving

Queue storage can fall back locally when MemoryStore is unavailable, which keeps matchmaking systems testable during Studio development.

Progression uses registries and appliers

Stands, cosmetics, pets, perks, rewards, and rig application are represented through modules instead of being buried in UI scripts.

Client UX includes recovery paths

Mobile controls, HUD watchdogs, resolution scaling, spectator UI, reward presentation, and VFX controllers support real player conditions.

Remote Architecture

A multiplayer surface organized by responsibility.

The remotes are not treated as a dumping ground. They are split by gameplay domain so combat, progression, inventory, queueing, party flow, and competitive match state can evolve without becoming impossible to reason about.
